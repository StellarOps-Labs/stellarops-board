# Issue Readiness

A contributor-ready issue is specific enough for a contributor to understand the work, make a plan, and know when the work is complete.

## Required Signals

- Problem statement: explains the pain or missing behavior.
- Expected behavior: describes the desired result.
- Files to update: points contributors toward likely code or docs.
- Acceptance criteria: lists objective review checks.
- Difficulty level: helps contributors self-select.
- Useful labels: communicates area, type, and readiness.
- Stellar/Soroban relevance: explains ecosystem value.

## Scoring

StellarOps Board gives each issue a 0 to 100 score. High-scoring issues are ready, middle scores need work, and low scores are weak. Missing fields produce suggestions that maintainers can use before publishing contributor calls.

## Maintainer Guidance

Strong issues reduce review churn. Add acceptance criteria before asking for help, include likely files when possible, and be explicit about whether the work affects Stellar, Soroban, wallets, developer docs, tests, or frontend flows.
